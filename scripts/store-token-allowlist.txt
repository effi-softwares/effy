# store-token-allowlist.txt — the only places the retired token "store" may survive.
#
# Consumed by scripts/verify-no-store.sh (008-shop-naming-unification, SC-001).
# One extended-regex per line, matched against a `path:line:text` grep hit.
# Every pattern MUST carry a comment naming its category letter and its reason.
#
# Categories (specs/008-shop-naming-unification/contracts/naming.contract.md § 3):
#   (a) TanStack Store / client-state terminology — a library name, not the audience
#   (b) the customer "storefront" — the product model's own word
#   (c) AWS "Parameter Store" — an AWS product name
#   (d) the English verb, and unrelated compounds
#   (e) historical record — text that would be FALSIFIED by rewriting it
#   (f) meta — artifacts describing the rename itself (path-excluded in the guard, not here)
#
# The guard greps for the SUBSTRING "store", case-insensitively — not a word boundary. That is
# deliberate: `\bstore` would miss `toStoreRoles`, `isStoreManager`, and `getStoreStaff`, which are
# exactly the misses that matter. The cost is noise (`restore`, `truststore`, `zipStoreBase`), which
# is what the (d) patterns below buy off. False positives are cheap; false negatives defeat SC-001.
#
# Adding a line here is a deliberate, reviewable act. A pattern broad enough to swallow a missed
# rename silently defeats SC-001. Never add a bare `stores?`.

#
# 011 built `apps/customer-web`: the customer STOREFRONT. Customers browse "the store" — a single
# Effy brand. That is the product model's own word (platform-brief / CLAUDE.md) and it is a
# DIFFERENT CONCEPT from the `shop` audience, which names the hidden internal fulfilment nodes a
# customer never sees. Renaming customer-facing copy to "shop" would not fix a naming split; it
# would CREATE one, by using the audience's word for something that is not the audience.
